# 实习日志（7月12日 - 7月17日）

## 7月12日（周六）

### 实习内容
1. 接到实习任务：基于 Arduino UNO 实现平衡小车的控制程序。
2. 在 GitHub 上找到开源项目 [lxbme/arduino-balance-car](https://github.com/lxbme/arduino-balance-car)，clone 到本地。
3. 通读原项目代码 `Minibalance_for_Arduino_Final.ino`，理解整体架构：
   - 三环控制：直立环（PD，5ms）、速度环（PI，40ms）、转向环（20ms）
   - 卡尔曼滤波融合 MPU6050 数据
   - 单编码器模式（仅左轮使用外部中断 0）
4. 阅读 `平衡小车原理图.pdf`，对照代码理解硬件接线方案。

### 收获体会
- 第一次接触平衡小车项目，对 PID 控制理论有了直观认识。之前只在课本上学过 PID 公式，实际看到代码中如何将其落地——直立环用 PD（不需要 I 因为角度不需要消除稳态误差），速度环用 PI（需要积分消除速度累积误差），这种差异化的设计让我理解了"不同控制目标适用不同控制策略"。
- 学会了阅读开源硬件项目的思路：先看原理图理解硬件拓扑，再对照代码理解引脚分配和控制流程。
- 原作者的项目结构清晰，注释到位，是很值得学习代码风格的开源项目。

## 7月13日（周日）

### 实习内容
1. 开始编写适配版代码 `Minibalance_Adapted.ino`，主要目标：
   - 将单编码器扩展为双编码器模式（左右轮独立测速）
   - 添加详细中文注释，方便后续调试和理解
   - 移除原代码中不使用的 ESP32 I2C 通信模块，精简为纯平衡控制
2. 研究右编码器的接线方案。原项目因 Arduino UNO Pin 4 不支持 PinChangeInt 而放弃右编码器，决定先尝试使用 PinChangeInt 库启用 Pin 4 的引脚变化中断。
3. 规划了完整的硬件接线方案，在代码头注释中详细记录。

### 收获体会
- 代码注释的重要性体感很深。原著代码虽好，但关键逻辑没有中文说明，调试时来回翻看很费时间。写适配版时我尽量把每段逻辑的中文含义写清楚，这对后续调参阶段的帮助会很大。
- 理解了一个嵌入式开发中常见的问题：硬件引脚资源是有限的。Arduino UNO 只有两个外部中断（Pin 2 和 Pin 3），Pin 3 被按键占用，所以右编码器不能像左编码器那样直接使用外部中断，必须另寻方案。
- 学会了使用 `INPUT_PULLUP` 模式的时机——对于编码器这类开漏输出的传感器，内部上拉可以避免引脚浮空导致的误触发。

## 7月14日（周一）

### 实习内容
1. 烧录适配版代码到 Arduino UNO，上电测试。遇到三个关键问题并逐一排查解决：

   **问题一：PinChangeInt 导致系统反复崩溃**
   - 现象：启用 PinChangeInt 后系统不断重启，串口输出乱码。
   - 排查：查阅 ATmega328P 数据手册，发现 Pin 4 属于 PORTD 端口，与硬件串口（TX=Pin1, RX=Pin0）共用同一端口。PinChangeInt 在 PORTD 上的中断处理与串口中断产生冲突。
   - 解决：放弃 PinChangeInt，改为在 5ms 定时中断（MsTimer2）中轮询检测 Pin 4 电平变化。5ms（200Hz）采样率对平衡车低速编码器足够。

   **问题二：I2C 总线锁死**
   - 现象：上电后 I2C 扫描找不到 MPU6050（0x68 无应答），但拔电重启后偶尔正常。
   - 排查：MPU6050 上次通信异常时将 SDA 线拉低，Arduino Wire 库无法自动恢复。
   - 解决：编写 `I2C_Recover()` 函数，在 I2C 初始化前手动模拟 9 个时钟脉冲 + 停止条件来释放 SDA 线。

   **问题三：编码器浮空误计数**
   - 现象：编码器频繁出现随机跳变，速度环积分快速累积导致抖动。
   - 解决：编码器引脚统一使用 `INPUT_PULLUP` 初始化，用 Arduino 内部上拉电阻（约 20KΩ-50KΩ）确保空闲时电平稳定。

2. 添加了启动诊断流程：I2C 扫描、分步初始化、错误提示与 LED 闪烁报警，方便快速定位硬件问题。

### 收获体会
- 今天是最累但收获最大的一天。三个问题每个都花了一两个小时排查，深刻体会到"硬件调试和纯软件调试完全不同"——软件 bug 看日志就能定位，硬件问题可能是不稳定的，时好时坏，排查难度大很多。
- 学到了一个重要的方法论：遇到硬件级问题先看芯片数据手册（datasheet），而不是盲目试错。PinChangeInt 和串口冲突这个问题，如果不查 ATmega328P 的端口映射表，可能永远想不到原因。
- I2C 总线恢复这个技巧是之前完全不知道的知识点。查阅资料才知道这是 I2C 协议的经典问题——从设备在主机读取时卡住，把 SDA 拉死不释放。手动发时钟脉冲是最标准的恢复方式。
- 今天接触了很多低层知识（端口寄存器、中断向量、I2C 电气特性），虽然过程艰难但感觉对嵌入式的理解上了一个台阶。

## 7月15日（周二）

### 实习内容
1. 正式进入 PID 参数调试阶段。参考了 B 站视频「【平衡小车PID】直立环+速度环完整调参过程」，学习了系统性的调参方法论。
2. 进行了三轮参数实验：

   | 轮次 | KP | KD | 现象 |
   |------|-----|-----|------|
   | 初始 | 15.0 | 0.4 | 摆动幅度大，有发散趋势 |
   | 第一轮 | 15.0 | 1.5 | 高频抖动严重，电机发热 |
   | 第二轮 | 8.0 | 0.8 | 响应太慢，无法维持站立 |
   | **最终** | **11.0** | **0.8** | **起步温和，稳定站立** |

3. 速度环参数同步调整：VKP 从 2.0→2.5，VKI 从 0.01→0.011。
4. 设置 PWM_MAX=180 限制最大输出，保护电机。
5. 确定 TARGET_ANGLE（机械中值）为 -1.0。

### 收获体会
- 调参过程让我真正理解了 PID 三个参数的含义，而不只是背公式：
  - **KP（比例）**：决定"多用力"去纠正偏差。太大会过冲振荡，太小会响应太慢站不住。
  - **KD（微分）**：决定"阻尼多强"去抑制振荡。太大高频抖，太小抑制不住。
  - **KI（积分）**：消除稳态误差。速度环才需要，加到直立环反而会引入低频振荡。
- 调参一定要有方法、有记录，不能凭感觉乱改。每次改动只改一个参数、记录现象，才能建立起"参数→效果"的对应关系。
- B 站视频里讲到"先调直立环（关速度环），站稳了再加速度环"，这个方法论非常关键。如果两个环一起调，根本分不清问题是哪个环引起的。
- 平衡小车最直观的感受是：参数差一点，车子就是站不住。这种"差之毫厘谬以千里"的体验让我对控制理论有了更深的敬畏。
